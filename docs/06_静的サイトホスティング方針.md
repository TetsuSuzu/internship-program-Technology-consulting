# 06. 静的サイトホスティング方針

## 前提・制約
- 学生個人のAWS/クラウドアカウントは用意しない
- 企業GitHub環境は public 禁止 → private → **private での GitHub Pages は有料**
- 参加人数：10〜20名
- クラウド（Azure/AWS）の発行可否：**未確定**

---

## 設計の肝：作る部分と公開部分を分離する

```
[共通パート（クラウド非依存・今すぐ作れる）]
  Codespacesで index.html を作る → 手元プレビューで「動いた！」

        ↓ ここから先だけ環境で分岐（後から確定でOK）

[公開モジュール（どれか1つ）]
  ├─ Azure Blob 静的サイト
  ├─ AWS S3 静的サイト
  └─ Codespacesプレビューのみ ← クラウド未確定でもこれで成立（保険）
```

→ 「公開」は最後の15〜20分の1モジュール。ここが未確定でも全体設計は止まらない。
**Codespacesプレビュー版を必ず用意**し、当日クラウドが間に合わなくても破綻しないようにする。

---

## 各方式の比較（10〜20名）

| 観点 | AWS S3 | Azure Blob | Codespacesプレビュー |
|------|--------|-----------|--------------------|
| 学生アカウント不要 | ◯（IAM配布） | ◯（SAS配布） | ◯ |
| 本物のクラウド体験 | ◎ | ◎ | △（手元のみ） |
| 公開ガードレール | **要調整（最難関）** | 影響を受けにくい傾向 | 影響なし |
| 講師の事前準備 | 重い（人数分IAM+バケット→要スクリプト） | 中（1アカウント+サブフォルダ） | 軽い |
| 初心者の難易度 | 高め | 中 | 低 |
| コスト | ほぼ無料 | ほぼ無料 | 無料枠 |

- **AWS S3**：`Block Public Access` の公開許可調整が必須。10〜20名はCLI/スクリプトで一括作成推奨。
- **Azure Blob**：1ストレージアカウント＋学生ごとサブフォルダでURLを分けられる。準備が比較的軽い。
- 静的サイトは Azure の方が公開ガードレールの影響を受けにくい傾向。

---

## 推奨アクション
1. クラウド非依存の共通パート（作る→プレビュー）の教材を先に作る ← 無駄にならない（→ `handson/`）
2. クラウドが Azure で確定したら Azure 公開手順を差し込む
3. 間に合わなければ Codespacesプレビューで実施（就業体験として十分成立）

## 社内で確認すべきこと
- [ ] AWS / Azure どちらのアカウントを発行できるか
- [ ] 公開（匿名読み取り）が許可されるか（S3が最難関）
- [ ] 学生への権限配布方法（IAMユーザー / SASトークン）がポリシー上OKか
